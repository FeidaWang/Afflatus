# prompts/ — v1.5 定时任务提示词库

> 每个 v1.5 模块的定时任务提示词正式文本。设计规范对应 ROADMAP §5c-6。
> 文件结构统一：用途/频率 → System Prompt（英文为 API 正本）→ Run Payload 结构 → 输出 JSON Schema → 中文对照要点。

## 五条硬规则（所有模块共用）

1. **system/run 拆分**
   角色定义、不可逾越的红线、输出 schema 全部放进固定不变的 system prompt——逐字节稳定才能吃到 prompt caching。每次运行只有 run payload（状态 + 当日数据）在变。**不要**为了"顺手"把日期、行情数字写进 system prompt。

2. **强制 JSON 输出**
   模型输出的机器段必须是单个 JSON 对象（无 markdown 围栏、无前后闲话），脚本直接 `JSON.parse` 落盘。展示用的复盘/推理文案是 JSON 内部的限长字段，不是自由文本。解析失败的 run 视为失败，脚本重试一次后放弃并记日志——不要人肉修输出。

3. **状态外置**
   模型零会话记忆。上一轮发生了什么，全部以结构化状态（账本、战绩、拒单记录）注入本轮 payload。任何"记得昨天"的设计都是 bug。

4. **事实纪律**
   模型只允许引用 payload 注入的数据。禁止凭训练记忆报价格、新闻、比赛结果、模型参数。payload 里没有的信息 = 不存在；缺数据时在 `notes` 字段声明缺什么，而不是编一个。所有判断带 `confidence`（0–1），低置信度自动降权/不执行。

5. **长度硬上限**
   复盘 ≤300 字（中）/≤120 words（英）；单条推理 ≤120 字；新闻摘要注入前先由脚本截断（每条 ≤40 词）。超限即截断，防 token 膨胀。

## 数据预消化原则

Fable 只做决策推理，不做数据搬运。注入前由脚本完成：指标计算（均线/RSI/涨跌幅等纯算术）、新闻去重与截断、日历判断（是否交易日/比赛日）。**一个数字模型算得出来 ≠ 应该让模型算。**
