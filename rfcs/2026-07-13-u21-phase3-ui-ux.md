## 1. 证据明细

### 1.1 间距：8px 网格合规率低

```
总计 566 处 padding/margin/gap 像素值
├─ 8px 网格（含 0/8/16/24...）   81 处  14.3%
├─ 4px 网格但非 8px（4/12/20...）100 处  17.7%
└─ 完全不在 4px 网格上           385 处  68.0%
   高频漏网值：6px×66, 7px×64, 5px×56, 10px×44, 3px×42, 9px×29, 14px×16
```

7788 行 append-only 层叠里每次「差不多就行」地手写一个像素值，68% 不在网格上是这种写法的必然结果，不是哪一次改动的错——所以修法也不该是逐行改数字，是给一套间距 token 之后新代码只准用 token（见 §2.2），存量值不追溯修改（同 Phase 1 D1 的「渐进接管、不重写」纪律）。

### 1.2 对比度：token 系统本身基本达标，但有一个真失败项

用 WCAG 相对亮度公式对 `:root` 核心 token 逐个跑了一遍（背景统一取 `--bg:#04060a`，最不利情况——实际卡片/面板背景更亮，对比度只会更好）：

| Token / 颜色 | 对比度 | WCAG AA（正文 4.5:1） |
| --- | --- | --- |
| `--ink`（正文）#e4e8f0 | 16.51:1 | 通过 |
| `--dim`（次要文字）#69748c | **4.32:1** | **不通过**（差 0.18，非大字号时不合规） |
| `--cyan`（强调）#8db4c0 | 9.10:1 | 通过 |
| `--warm`（强调橙）#e8b380 | 10.81:1 | 通过 |
| `#edf5ff` `#9ae5ff` `#5dff9d` `#ffd45d`（高频强调色，各用 13/13/10/6 次） | 14.5–18.5:1 | 通过 |

`--dim` 是这套 token 里唯一不达标的，但它恰恰是使用面最广的一个——`.nav-clock span`、`.top-stat`、大部分标签/时间戳/次要说明文字都在用，且字号集中在 9–11px（WCAG 大字号豁免线是 18px/14px-bold，这批文字全部不豁免）。这是一处**真的、可修的合规缺口**，不是审美偏好问题。

另需人工复核（本次代码审计给不出结论）：`opacity:.3x`–`.4x` 一共 12 处，用在文字上时会把底色和背景做透明混合，实际对比度取决于混合后的合成色，不是简单查 hex 表能算的——Phase 3 动码时逐个截图核对，或者干脆替换成一个专门算好对比度的 `--dim-faint` token（比调 opacity 更可控，也更符合 §2.1 的 token 化方向）。

### 1.3 Restraint：48 处 backdrop-filter + 5 处焦点样式移除

- `backdrop-filter` 48 处——Lesson 2.7 已从性能角度点过名（移动端 GPU 大户），Restraint 维度是同一组证据的另一个提问角度：「这 48 处玻璃拟态都有人在用吗」。建议 Phase 3 动码时先列出全部 48 处的用途和触发条件，删掉「装饰性、无状态变化含义」的那些——**红线：删一个元素页面不塌 = 该删**，不是本文越俎代庖去猜，交给站主逐条判断，但审计出这个数字本身就是有价值的信息。
- `outline:none`/`outline:0` 共 5 处，其中 3 处（`.notebook-login input`——基础规则去掉默认环、`:focus` 状态另有 `box-shadow` 发光 + `border-color` 变化补上；`.afpc-input:focus` 同款写法）**是合格的自定义焦点样式，不是问题**。真正的缺口是 2 处：`.mobile-weapon-form select`、`.starmap-controls input[type="range"]`——去掉了默认焦点环，且全文没找到对应的替代视觉反馈。样本小，Phase 3 动码时顺手修，不需要专项。
- 全站仅 4 处 `:focus`/`:focus-visible` 规则（7788 行样式表里）；同时 `cursor:none` 出现 14 次（自定义十字准星/HUD 光标是站主的美术设计选择，合理），但这意味着**键盘用户完全依赖这 4 处焦点样式定位**——密度明显偏低，是本文认为最值得在 Phase 3 里系统补的一项（见 §2.3）。

### 1.4 触控目标尺寸：移动端两处低于 44px 建议下限

`.notebook-login input/button{min-height:26px}`、`#commandModeBtn{height:28px}`——iOS HIG 与 WCAG 2.5.5 都建议交互元素触控目标 ≥44×44px（AA 是「建议」级，AAA 才是强制，但移动端命中率的实测数据一贯支持这条经验值）。样本同样小（2 处），Phase 3 动码时顺手放大到 ≥44px 或用 `padding` 撑够可点击区域（不必强制视觉尺寸变大，触控热区可以比视觉尺寸大）。

### 1.5 视觉层级：本次审计给不出真实分

Lesson 2.8 的视觉层级维度要求「3 秒内看出先看哪、后看哪」，这个判断依赖真实渲染画面，代码审计（选择器数量、DOM 嵌套深度）只能做非常粗糙的代理，参考价值有限，本文不强行编数字——§0 表格里给的 18 分是基于历史记录（U8/U13/U14 已做过整治）的保守估计，**不是本次新证据得出的**，站主真机复核时这一维度的分数请以真机为准，不要采信本文。

## 2. `@layer tokens` 设计令牌方案（不引 Tailwind，Phase 1 §1.5 已裁决）

Phase 1 已经建好空的 `@layer tokens {}` 骨架（`src/styles.css` 顶部）。Phase 3 要填的不是「重新设计一套配色」，是**把已经在用的值收编成可复用 token**——103 个唯一 hex 色值里，真正的设计意图可能只有一二十种，其余是同一个颜色的浮点误差版本（截图对比、复制粘贴累积的噪音）。

### 2.1 颜色 token：在现有 `:root` 基础上扩展，不推倒重来

现状 `:root` 已有 `--bg`/`--ink`/`--dim`/`--faint`/`--cyan`/`--warm` 六个基础 token 和第二个 `:root` 块里的 `--hud-*` 五个——这个体系本身是对的，问题是覆盖不全：`var(--...)` 总用量 146 次里，色彩相关的 token（`--dim`/`--ink`/`--warm`/`--cyan`/`--weapon-*`/`--forge`/`--line` 等）约占三分之二，其余是布局尺寸类 token（`--hud-combat-h`/`--radar-size` 等，不在本次颜色审计范围内）；而全文仍有 196 处颜色是直接写 hex/rgba、完全绕开 token 系统的。方案：

```css
@layer tokens {
  :root {
    /* 现有 token 原样保留，不改值（--dim 的合规问题单独处理，见下） */

    /* 新增：--dim 的 AA 合规替身，用于实际需要阅读的小字号文字 */
    --dim-aa: #6c7790; /* 对 --bg 对比度 4.52:1，压线通过，比 --dim 只亮约 3%（数值算出，未真机比对色差是否可辨） */
    /* --dim 保留给「明确非阅读用途」的场景（如已经有其他视觉线索的装饰性标签） */

    /* 高频强调色收编（各自在源码中已出现 6–13 次，直接复用现有 hex，只是给名字） */
    --accent-blue: #9ae5ff;
    --accent-mint: #5dff9d;
    --accent-amber: #ffd45d;
    --accent-highlight: #edf5ff;
  }
}
```

迁移策略与 Phase 1 §2.1 的双语字段一致：**新写的 CSS 一律用 token，存量 196 处硬编码不追溯替换**（7788 行逐条替换的风险收益比很差，且本文没有能力真机验证每一处替换后的视觉效果）。`scripts/check-no-new-important.mjs` 的模式可以复用一份思路，写一个「新增硬编码颜色计数不许超基线」的姊妹脚本，把 §1.2 的合规意图变成可持续执行的纪律，而不是一次性文档。

### 2.2 间距 token：8px 网格量表

```css
@layer tokens {
  :root {
    --space-1: 4px;  --space-2: 8px;  --space-3: 12px; --space-4: 16px;
    --space-5: 24px; --space-6: 32px; --space-7: 48px; --space-8: 64px;
  }
}
```

同样只约束新代码。§1.1 的 68% 离网格值不做批量替换——很多是历史上「肉眼调出来刚好好看」的值，机械替换成最近的网格值有实际视觉回归风险，这正是「需要真机验证不能盲改」的典型情况。

### 2.3 焦点样式 token：补 §1.3 发现的覆盖缺口

```css
@layer tokens {
  :root {
    --focus-ring: 0 0 0 2px var(--bg), 0 0 0 4px var(--accent-blue);
  }
}
@layer components {
  :is(button, a, input, select, [tabindex]):focus-visible {
    outline: none;
    box-shadow: var(--focus-ring);
  }
}
```

用 `:focus-visible` 而非 `:focus`——键盘 tab 导航时显示，鼠标点击（配合站主自定义的 `cursor:none` 十字准星）不显示，两种输入方式互不干扰。这一条直接解决 §1.3「全站仅 4 处焦点样式」的覆盖缺口，且是一条通用规则，不需要逐个组件手改。

## 3. 微交互 / 自适应布局建议（方向性，非逐行代码）

- **微交互只服务状态变化**（Lesson 2.8 红线原文）：现有 `sysBreath` 呼吸动画、`energy-bar` 过渡这类「状态在变、用动效强化感知」的用法是对的，不建议新增。真要加，先问「删掉这个动效，用户会不会看漏一个状态变化」，答案是否定就不加。
- **自适应布局**：现状 `--hud-cols` 等断点值已经是响应式思路，Phase 3 不需要引入新的布局范式（如 container queries）——除非 Phase 2 的移动端 HUD 简化（course Lesson 2.7 那条）之后还留有断点腰斩的问题，那属于 Phase 2 收尾而非 Phase 3 新增范围。
- **暗色模式**：站主已经是全站深色科幻主题，「暗色模式」这个课题在这里的正确翻译是"tokens 化 + 对比度合规"（本文 §1.2/§2.1），不是新增一套亮色主题——课程原始 prompt 提到的「dark mode implementation」在此语境下已经完成，不重复造轮子。

## 4. 组件结构与红线

- **红线（Lesson 2.8 原话）**：删一个元素页面不塌 = 该删；新特效先问「谁需要它」。§1.3 的 48 处 `backdrop-filter` 清单交给站主逐条判断，本文不代为决定删哪些——克制维度的分数能不能从 20 提到 25，取决于这个清单的执行力度，不是加什么新代码。
- **组件重排**：本次审计没有发现需要立项重排的组件级问题（对比 Phase 1 数据层审计发现了 4 个真反模式），Phase 3 更像是「系统化收敛」而非「结构性重来」——这本身也是一个结论：不要因为课程大纲写了「组件结构与直觉导航重排」就一定要动结构，现状的组件划分（`.book`/`.gauge`/`.hud-*` 等）没有暴露出真实问题。
