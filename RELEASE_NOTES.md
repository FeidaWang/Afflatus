### 观星台立盘：修复时区/经纬度未真正隐藏的老 bug + 城市选择器改为「省份/国家 → 城市」两级下拉 — 2026-07-10

**背景**：站主截图反馈"出生地时区和经纬度依然显示在页面上"。排查发现是个上一次改动之前就存在的老 bug——`.fld-manual{display:flex}` 这条规则的优先级意外盖过了元素的 `hidden` 属性（两者 CSS 优先级相同，作者样式表后于浏览器默认样式表生效，author 规则赢），导致"找不到城市？手动填写"折叠区从城市选择器上线那次起就从来没有真正隐藏过，只是视觉验收一直标注"待真人浏览器验收"没有抓到。修复：加一行 `.fld-manual[hidden]{display:none}`，现在会真正隐藏。

同一条反馈里，站主还提到城市列表"太齐全拉开篇幅太大"，要求国内城市按省市分类、海外城市按国家分类重新整理。原来的实现是一个绑定 `<datalist>` 的搜索框，526 个城市全部拍在一个列表里——`<datalist>` 原生不支持分组，没法做到"分类"。改为两级下拉：先选省份/国家，再从这个省份/国家的城市里选——`<select>` 原生支持 `<optgroup>`，真正做到了视觉分组，也不再一次性铺开几百个选项。

**数据补充**（`src/lib/cities.js`）：356 个中国城市新增 `province` 字段（标准行政区划归属，34 个省级单位，公开事实型分类数据，不是需要数值校验的坐标测量，用脚本批量生成并做了"每个城市都必须映射到某个省份、映射结果覆盖全部 356 个城市"的完整性校验，跑出一处遗漏—连江—已补上）；170 个境外城市新增 `country`/`countryZh` 字段（137 个国家/地区，同样做了完整性校验，无遗漏无多余）。

**`src/lib/cityPicker.js` 重写**：`allCityOptions()`/`findCityByLabel()`（扁平搜索用）替换为 `allRegions()`（返回省份列表+国家列表）/`citiesInRegion()`（某个省份或国家下的城市）/`findCityInRegion()`（在已选区域内按城市名查找）。`horoscope.html` 两个表单的城市输入框换成"出生地区"+"出生城市"两个级联下拉（选地区后城市下拉才启用并填入该地区的城市）；`horoscope.js` 新的 `wireRegionCityPicker()` 选中城市后照旧回填隐藏的 `bTz`/`bLat`/`bLon`（或 `sTz`）字段，`parseBirth()`/`normalizeBirthToCST()` 零改动。

**验证**：487/487 vitest（`cityPicker.test.js` 全面重写，12 例，含"356 个城市 province 字段全覆盖""34 个省级单位数量准确""170 个城市 country 字段全覆盖"等完整性断言）；`tsc --noEmit` 干净；scratch `vite build` 确认星历分包不受影响，主 `horoscope-*.js` 151KB→167KB（新增省份/国家分类数据+级联选择逻辑）。**视觉（两级下拉的实际排版/宽度）仍待真人浏览器验收——这也是上一条老 bug 被漏掉的原因，这次会更留意。**

### 观星台：MBTI 扩容 + 新增智商/情商娱乐速测 + 一键清空隐私 + 立盘表单再精简 — 2026-07-10

**背景**：站主原话——"把页面上的 MBTI 测试做的更完整更具娱乐性，再加入一个业界最先进的智商测试和情商测试。两个用户的立盘测试都需要配置一键清空效果，保护隐私……出生地时区，纬度和经度，找不到城市和手动填写区域全部删除，让输入页面尽可能的保持简洁"。一次请求，四件事。

**"业界最先进"的诚实处理**：没有照字面做——网页娱乐小测不可能是"业界最先进的智商/情商测试"，这么写就是失实宣传。做法是原创、认真设计的智商风格谜题测和情商风格情境测，延续本页 MBTI 克隆早已定下的姿态（`persona.js` 头部原有的"与 Myers-Briggs Type Indicator® 无关联，仅供娱乐"声明），在两个新测试的页面文案里同样直接写明"不是经过验证/临床校准的测评工具"，没有做业界最先进的宣称。

**MBTI 速测扩容**（`src/lib/persona.js`）：6 题/轴 → 8 题/轴，共 24→32 题；新增 `PERSONA_MATCH`——16 个类型各配 2 个"合得来"类型 + 1 个"易摩擦"类型（民间 MBTI 配对梗，非严肃工具，结果页同样娱乐化呈现）；结果卡新增「保存结果卡」按钮，`shareCard.js` 新增 `'persona'` 卡片类型（四轴进度条 + 类型名）。

**新增 `src/lib/logicQuiz.js`**（智商风格速测）：原创 16 题（数列/类比/找不同/简单逻辑），4 选 1，答对数映射到 4 档娱乐评语，另附一个 70-150 区间的"娱乐分数"（明确标注"不是真实智商分数"）。

**新增 `src/lib/eqQuiz.js`**（情商风格速测）：原创 15 题生活情境，覆盖 Goleman 公开框架的 5 个维度（自我觉察/自我调节/内驱力/共情力/社交技巧，公开概念非商标），每题 4 个回应选项按 0-3 打分；结果用已有的 `astroChart.js` `renderRadar()` 五维雷达图渲染（与 L2 五维雷达同一渲染器，零新增图形代码）。

**一键清空隐私**（`horoscope.js`/`horoscope.html`/`horoscope.css`）：hero 区新增"⌫ 清空本机保存的全部数据"按钮，点击二次确认后按 `afflatus-horo:` 前缀批量清除 localStorage（命盘、连续签到、关系册、MBTI/智商/情商测试结果全覆盖，前缀式清除意味着未来新增测试无需再改这段代码），随后整页刷新。

**立盘表单再精简**：移除"找不到城市？手动填写"折叠按钮（`horoscope.html` 两处），原时区下拉+纬度+经度字段本身保留在 DOM 中（`hidden`，无 UI 入口），只是不再对用户可见——`parseBirth()`/`fillForm()`/`wireCityInput()` 零改动，城市选择器照常往这些隐藏字段里写值。夏令时勾选框未被要求删除，继续常驻可见（历史夏令时规则是城市数据库唯一没法自动推断的一项）。

**验证**：484/484 vitest（新增 `logicQuiz.test.js` 7 例、`eqQuiz.test.js` 7 例，`persona.test.js` 同步更新为 32 题/8 轴 + match/friction 完整性断言）；`tsc --noEmit` 干净；scratch `vite build` 确认星历库 `astroPlanets-*.js` 依旧独立分包（46.7KB 不受影响），主 `horoscope-*.js` 134KB→151KB（新增两套题库+结果渲染）。**视觉（三个测试的排版、清空按钮位置、结果卡样式）仍待真人浏览器验收**。

### 观星台立盘：出生城市选择器，替换手动时区+经纬度 — 2026-07-10

**背景**：站主原话"立盘加入出生地具体城市选择以替换出生地时区和经纬度，节省布局空间"。改前"我"的表单要填 4 个独立字段（时区下拉 + 夏令时勾选 + 纬度数字 + 经度数字），"对方"表单要填 2 个（时区+夏令时）——大多数用户既不知道自己出生地精确经纬度，也不确定当时是否实行夏令时。

**数据来源与核实**（本项目对手打地理/科学数据一贯谨慎——同一套纪律曾让站点选择引入 astronomy-engine 而非手打行星星历系数，这次同理）：
- 中国 356 个地级市：抓取一个公开数据集，再用第二个独立的（区县级、带省份标签的）数据集交叉核对，外加机械重复坐标检测——抓出并修正了 5 个源数据本身的错误（中山、大同、甘南被错误标成了东北地区坐标；兴安与大兴安岭、资阳与益阳两对被源数据误标成完全相同的坐标）。修正值均有 WebSearch 独立信源支持，不是凭记忆估的。
- 全球 169 个主要城市：国家首都坐标表 + 针对纽约/东京/香港/新加坡/迪拜等主要枢纽城市的定向检索，同样交叉核对，过程中抓到并修正了源表里一个偏差约 240 公里的布宜诺斯艾利斯坐标错误。**已知缺口**：目标是约 200 个，实际拿到 169 个（+ 后续补的东京 = 170）——多数公开的"城市+时区"数据集在抓取时被截断，凑数据宁可缺项也不编造，如实告知。
- `src/lib/cities.js`：新增数据模块，中国城市不带 `utcOffset` 字段——特意如此，因为留空会走 `horoscope.js` 里已有的"未指定 · 按中国处理"路径，那条路径会自动修正中国自己 1986–1991 年的夏令时窗口（`bazi.js` 的 `CN_DST_WINDOWS`），比手动写死 `{utcOffset:8, dst:false}` 更准——后者会跳过那次历史修正。

**UI 改动**：`horoscope.html` 两个表单的时区下拉+纬度+经度字段，默认收进一个"手动填写"折叠区（点"找不到城市？手动填写"展开，原字段一个没删，全部保留）；取而代之的默认路径是一个 `<input list>` 绑定 `<datalist>` 的可搜索城市输入框（中英文都能搜，"东京"/"Tokyo"都能命中），选中后自动回填隐藏的时区下拉+纬度+经度（复用现有 `parseBirth()`/`normalizeBirthToCST()` 逻辑，零改动）。夏令时勾选框依然常驻可见——历史夏令时规则复杂到没法用一个城市数据库自动推断，这一项继续交给用户自己判断，是唯一保留在默认视图里的"精确性"字段。"我"的表单 4 字段变 2 个（城市+夏令时），"对方"表单 2 字段变 2 个（但去掉了时区下拉，换成信息量更大的城市输入）。

**新增 `src/lib/cityPicker.js`**：纯函数层（`allCityOptions()`/`findCityByLabel()`），与原始数据分离，独立可测。

**验证**：469/469 vitest（新增 `cityPicker.test.js` 9 例，含"5 个已知错误条目坐标已修正""全球城市 UTC 偏移值必须精确匹配现有 34 档时区下拉选项——否则选中城市后下拉会静默回退空值"两类关键断言）；`tsc --noEmit` 干净；scratch `vite build` 确认新增城市数据只增大主 `horoscope-*.js`（109KB→134KB，gzip 后 59KB），星历库 `astroPlanets-*.js` 分包不受影响。**视觉（城市输入框/手动折叠区排版）仍待真人浏览器验收**。

**streak 损失厌恶文案**（星语签卡片内，`.sd-streak-hint`）：streak<7 时显示"连续 N 天，还差 (7-N) 天解锁隐藏签面"的倒计时；streak≥7 后文案换成鼓励继续保持，而不是消失——文案会随进度真实变化，不是常驻的固定催促语。

**ICS 日历订阅**：缘分日历区新增"把本周最佳共处日加入我的日历"按钮，纯客户端生成一个单事件的 `.ics` 文件（`downloadICS()`，标准 VCALENDAR/VEVENT 格式，全天事件）供用户下载导入自己的日历 App——**不建后端订阅源，不发任何推送，提醒完全由用户自己的日历 App 负责**，字面执行白皮书"通知由用户自己的日历 App 发，我们零基建零打扰"。最佳日期取 7 天里 `dailyPull()` 分数最高的一天（`bestDayThisWeek()`，复用缘分日历已经在算的同一组分数，没有新计算逻辑）。

### V23 Phase 3 · 观星台三个 DAU 机制：双人今日天象 + 星语签 + 缘分日历 — 2026-07-09

**背景**：白皮书模块二点名的三个"每日回访理由"，全部要求"客户端确定性计算，同一天全球一致"，且明确要求日运计算不应该为了几个行星度数把整个 astronomy-engine（星历库）拖进首屏。

**每日行运数据管线**（新增，roadmap 模块四"复用 push-data.sh 管线"）：
- `scripts/gen-transits-daily.mjs`：纯 Node 脚本（不经过 vite/tsc），直接用已装的 astronomy-engine 算出当天太阳/月亮/水星/金星/火星/木星/土星的地心视黄经，写入 `public/transits-daily.json`（实测 248 字节，远低于 <2KB 预算）。刻意复制而非 import `src/lib/astroPlanets.ts` 里的同一个公式——那个文件的头部注释明确要求"只能被浏览器端的动态 import() 到达"，这里是另一个不相关的运行时（Node 脚本），复制一行公式比破坏那条纪律更干净。
- 新增每日定时任务 `horoscope-transits-daily`（06:38 本地时间，每日）：跑生成脚本 → `git add` 仅这一个文件 → 复用 `scripts/push-data.sh` 同款提交/拉取/推送流程。首次生成已手动跑过一次，`public/transits-daily.json` 随本次改动一并提交，站点从上线起就有数据，不用等到明早任务首次触发。

**双人今日天象**（`src/lib/dailyTransits.js`，纯函数，零星历依赖——今日行运数字从上面的 JSON 拿现成的，本命侧只用轻量的日月，两头都不需要客户端加载 astronomy-engine，这正是模块四那句"连星历库都不用为日运加载"的字面实现）：今日行运的 7 颗行星分别与两人的本命日月做相位比对，取最紧密的 ≤2 条生成"今日 XX 触动你/TA 的日/月：……"文案，接在合盘结果页已有的"今日引力"（V20 `dailyPull`）下方。

**每日星语签**（`src/lib/starDraw.js`）：确定性种子 = 日期 + 本人生辰字符串，签面是 28 宿（复用 `xiu.js` 现成的 `XIU28_ZH`）+ 12 星座（复用 `bazi.js`/`astroChart.js` 现成的星座名与符号）共 40 张卡的收集品，**卡面身份本身零新增文案**——只有"今天适合……"的行动建议文案库（12 条，纯行动向，无吉凶措辞）是新写的。复用页面已有的连续签到 `STREAK_KEY` 计数器：streak ≥7 天后，每天有 25% 概率抽到一个 4 张卡的隐藏池（"归途/破晓/满盈/静潮"），不是一到 7 天就永久解锁全部隐藏卡，保留"收集品"的抽卡感。

**缘分日历**：没有新写计算逻辑——直接把 V20 已有的 `dailyPull(me, other, dateStr)` 循环跑未来 7 个日期即可，因为它本来就是"确定性种子 = 日期+双方生辰"的每日打分函数。今天/明天显示真实分数，第 3-7 天在 UI 上锁定（虚线卡片 + "N 天后解锁"提示），**这是时间解锁不是付费解锁**——分数其实已经算出来了，只是不提前显示，白皮书原句"不做付费解锁，只做时间解锁"在这里是字面执行。

### V23 Phase 2 · 观星台「双盘四段式」合盘改造 + 关系册 + 分享卡升级 — 2026-07-09

### V23 Phase 1 · 观星台「单盘三层披露」改造 + astronomy-engine 接入 — 2026-07-09

**背景**：V23 白皮书（roadmap §7.10）把观星页从"一次性平铺全部信息"改造为 L1/L2/L3 三层渐进披露，同时裁决了 V21 遗留的行星星座技术选型（手写系数 vs 引星历库）。

**技术选型**：选用 `astronomy-engine`（npm，MIT，纯 JS，`^2.1.19`）而非手写 VSOP87 系数表——`src/lib/astro.js` 原有的 Sun/Moon 计算是手写系数，但为 8 颗行星（Mercury..Pluto）手打系数表属于"无法自证正确性的数据录入"，超出本项目一贯的验证纪律，故引入库。**关键坑**：`Astronomy.EclipticLongitude(body, date)` 返回的是**日心**黄经（读 `astronomy-engine` 源码确认，内部调用 `HelioVector`），对命盘毫无意义；正确调用是 `Astronomy.Ecliptic(Astronomy.GeoVector(body, date, true)).elon`（地心视黄经）。此坑通过一次全年 2026 扫描发现（初版实现显示水星偏离太阳最大达 178°，物理上不可能超过 ~28°），修正后写入 `astroPlanets.ts` 头部注释并做成永久回归测试（水星/金星最大距角断言）。

**三层结构**（`horoscope.html` / `src/pages/horoscope.js` / `public/styles/horoscope.css`）：
- **L1**（`l1HTML`，始终可见）：日月升大三 + 3-5 条双语人格标签，来自新文件 `src/lib/astroReadings.js` 的 `personalityTags()`（太阳特质/月亮需求/上升第一印象/五行风味短语库 + 三方组合"统一"标签）。
- **L2**（`l2HTML`，滚动进入视口后展开）：爱情/事业/沟通/能量/成长五维雷达图（`astroChart.js` 的 `renderRadar()` 纯函数 SVG 渲染），数值来自 `dimensionScores()`（基础分 50 + 三方星座加权 + 八字五行加成，clamp 到 [5,96]，三档文案）。用 `IntersectionObserver` 触发首次展开动画。
- **L3**（`renderL3Body` + `l3-toggle` 按钮，默认折叠）：PRO 专业星盘——轮盘（`renderWheel()`，整宫制，从上升星座起排，相位连线）+ 相位格网（`renderAspectGrid()`）。**`astronomy-engine` 仅在用户点击展开 L3 时才动态 `import('../lib/astroPlanets.ts')`**，不进首屏；已用生产构建验证：`astroPlanets-*.js`（约 46.7KB，含真实星历常量）是独立 chunk，未被主 `horoscope-*.js`（89KB）或全站共享 `main-*.js` 引入。

**已知未做项**（不阻塞本 Phase，记录在案）：L1 标签的"长按生成贴纸"交互；L3 术语区目前是静态术语列表，未做 hover tooltip。

**新增文件**：`src/lib/astroPlanets.ts`（首个非 cursor.ts 的 `.ts` 模块，因 `tsconfig` 的 `allowJs:false` 需要配套 `src/lib/astro.d.ts` 最小环境声明）、`src/lib/astroChart.js`（纯函数 SVG 渲染器，按白皮书要求保持 `.js`）、`src/lib/astroReadings.js`。

**验证**：435/435 vitest（新增 astroPlanets/astroChart/astroReadings 三个测试文件共 27 个用例，含水星距角物理不变量、轮盘上升点位置、相位格网对称性等结构性断言，非快照测试）；`tsc --noEmit` 干净；scratch `vite build` 确认动态 import 分包隔离。**视觉（L1/L2/L3 实际观感、雷达图/轮盘绘制效果）待真人浏览器验收**——沙盒仅能验证 DOM 结构与数值逻辑。

### V22 移动端全局修复（站主实机截图驱动）— 2026-07-06

**① 翻页箭头**：`public/page-turn.css` 的 `.page-turn-controls`（signal/arena/sectors/games/league/horoscope/serial 共用）在 `@media(max-width:760px)` 直接 `display:none`——覆盖导航已由顶部 nav 承担，箭头只会在窄屏遮挡内容。**首页是例外**：index.html 用的是完全独立的 `.route-arrows`/`.route-arrow` 实现（`src/styles.css`，因为要跟战斗 HUD 的位置数学联动），不吃 page-turn.css 的规则，另外单独在其 `@media(max-width:860px)` 块里把 `display` 从（此前一次"恢复移动端翻页"改动遗留的）`block!important` 改成 `none!important`。

- Hero 标题溢出：`content.js` 的中英文标题各硬编码 1–2 个 `<br>`，桌面宽容器下没问题，但移动端 clamp 字号下配合硬换行仍会在字内再次软换行（截图里 "I command the fleet" 撑爆）。修法：`@media(max-width:880px)` 新增 `.hero-title br{display:none}`，去掉硬换行交给自然换行接管。
- **HUD 死区（真正的 bug，不是"加个 CSS 藏起来"能糊弄过去的）**：`index.html` `<head>` 里有一段内联 `<style>`（`@media(max-width:560px)`），用 `#combatHud .hud-left,#combatHud .hud-center{display:none!important}` + `#combatHud .hud-panels{grid-template-areas:"pilot right"!important}` 把移动端战斗 HUD 砍成两块。这段代码写于（推测）某次更早的移动适配尝试，后来 `src/styles.css` 里 2026-06-14 那次"Mobile target-first HUD"重写引入了更完整的 4 面板 2 行网格（`@media(max-width:860px)`），但**从未删除**旧的内联块——由于 `#combatHud .hud-panels` 的 ID 前缀选择器优先级天生高于纯类选择器 `.hud-panels`，旧块在所有手机宽度下持续"赢"，把雷达和防御模块强制隐藏、grid-template-areas 又跟新网格的行数对不上，产生的就是截图里那块"无显示无交互"的死区。**修法是删除**那段过时的内联 `<style>`（index.html head），不是新增遮盖层——两套系统曾经在打架，现在只留新的一套。

**③ 移动端「去往其他页面」菜单**：`index.html` 早就有一个从未接线的 `.nav-menu-btn`（☰ 按钮 HTML 都写好了，`main.js` 里连事件监听都没有）；同时 `src/styles.css` 里有条更晚的 `@media(max-width:860px){.nav-menu-btn{display:none!important}}` 把它按下去了，二者叠加导致按钮实际从未出现过。修法：`main.js` 新增 `initNavSiteMenu()`，复用 `nav.js` 暴露的 `window.AfflatusSite` 页面表 + Labs 下拉同款"portal 到 body"弹层模式，构建一个列出全部 8 页的面板；`styles.css` 末尾追加最终覆盖把按钮在 ≤860px 显示出来。**只做了首页**——另外 7 个共用 `nav.js` 的页面本来就有随时可见的顶部链接 + 已工作的 Labs 下拉（点击触发，非纯 hover），风险评估后判断不需要再加同一个按钮；用户原话点名的是 "feida.au"（首页域名），首页也确实是唯一有死 UI 元素的地方。

**④ 小说页进度条缝隙（桌面+移动同源同修）**：`public/styles/serial.css` 里 `.reader-progress` 独立 `sticky top:0`、`.toolbar` 独立 `sticky top:8px`，两个互不relate的 sticky 元素分别贴边，中间露出 5–8px 正文——这不是"加点 margin"能稳定修的，因为 `.toolbar` 的实际渲染高度会随内容换行/断点变化。改法：`.toolbar` 改 `top:0`；`serial.html` 新增一小段 JS（`ResizeObserver` + resize 监听），把工具栏的**实测**高度写入 CSS 变量 `--toolbar-h`，`.reader-progress` 的 `top` 直接读这个变量——两条 sticky 栏永远贴合，不管工具栏是几行。

### V20.3.（S）晚子时日柱换日 bug——用户拿专业排盘网站交叉验证揪出来的真问题 — 2026-07-05

**问题**：用户在「问真八字」网站排了自己的盘（1992-02-23 23:26，乾造）比对，年柱月柱都对（壬申/壬寅），但日柱时柱不同——参考站给 庚午日 丙子时，本站给 己巳日 甲子时，质疑数据源有问题。

**根因**：不是数据源问题，是历法惯例选择——23 点出生正处「子时」，命理界对「子时是否换日」有两派：本站原实现是「早晚子时统一」（23:00-23:59 仍算当天），而问真八字等主流专业排盘站用的是「晚子时换日」（23:00-23:59 的日柱按次日算，时柱仍是子但用次日的日干起）。手算验证：把日期 +1 天重算，日柱变成 庚午——与参考站完全吻合。**这是本站此前偏离主流惯例的真实差异，值得修，不是伪 bug**。
